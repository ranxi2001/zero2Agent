面试感受：

和面的第一家中厂agent产品开发岗位有很大的差别，中厂更多是深入我的项目展开，这家公司没有深入问项目中AI开发相关的内容，偏向工程落地能力、基础原理和代码细节。问的基础内容也是简历上写的技能相关的（mysql，linux，redis，python）。而且问了一些实习的内容，但后续也没有继续深入问AI开发的细节。没有回答上的问题也没有给我讲解。

面试完过后更多的是困惑？不知道询问这些就可以筛选出他们所需要的求职者了吗？

虽然这家公司的岗位内容跟我的岗位预期的不太相同，从JD里其实也能看出公司内部可能对于AI开发也不太了解，几乎没写相关的技术栈，甚至python也写成了pvthon。

但是这次面试暴露出了我在八股方面的弱点，反思：对于简历上所写技能的八股还是要好好准备一下，后端开发的基础知识也需要好好掌握。

张口就有专业简历：找 Agent 方向实习，来求拷打，那我直说。 这份简历最大的问题是：你的项目经历和 Agent 方向的定位对上了，但没讲清楚。 看你的项目。"企业知识库智能问答系统"最对口，LangChain、RAG、向量数据库都有。但描述太短，看不出你对这个系统有多深的理解——RAG 的 chunk 策略怎么设计的，召回精度有没有评估过，向量数据库索引怎么建的，有没有碰到幻觉问题怎么处理的。这些才是面试官真正想听的，你现在的描述是"用了什么技术搭了什么系统"，这个层面几乎每个投 AI 实习的人都能写。 "基于 LangChain 的 Agent 至智搜索系统"名字挺响，但描述里只有工具调用、搜索功能的功能点，没有你对 Agent 架构本身的理解。ReAct 和 Function Call 的区别、多步推理的 trace 怎么调试、工具选择的 prompt 怎么设计，这些是面试会深挖的，简历里一点信号都没给。 另一个问题是个人优势那一栏。"熟悉大模型应用开发"和"掌握 AI 辅助开发能力"是今年所有投 AI 岗简历里最泛滥的表达，写了等于没写，反而让你的简历和其他人一模一样。 几个改法： 第一，把知识库问答系统的描述扩充，重点写你在 RAG 流程里做过哪些设计决策，遇到过什么问题怎么解的。哪怕一两个细节，也比现在的功能列表有说服力。 第二，Agent 系统那个项目，加一段你对 Agent 工作机制的理解，比如 prompt 设计思路、多工具场景下的路由逻辑，把你真正懂的东西写出来，让面试官看出你不是用 API 套模板的。 第三，"AI 辅助开发"这句话具体化：你用 AI 工具辅助了哪个项目的哪个环节，省了多少时间，这种具体描述比"掌握 AI 辅助开发"值钱十倍。 说个实话：Agent 方向实习现在很卷，对手很多是有论文或深度开源贡献的，如果你的项目都是课程级别的，进面之后基础原理和工程细节会被深问，这个工具改不了，得你自己补。 简历的表达这块，我用过微信小程序「聊出简历」，里面 AI 会追问你项目细节，专门针对"熟悉 AI 辅助开发"这类空话，把你真实做过的东西提炼出技术深度。这个痛点和你简历的问题挺契合的，你可以搜来试试。

# 哔哩哔哩大模型面试岗，我悟了！！！

> AIGCx小白入门记 | 05-03 18:00

**标签**：我的求职进度条

**来源**：https://www.nowcoder.com/discuss/880512077826187264

---

哔哩哔哩大模型面试岗，我悟了！！！
哔哩哔哩大模型面试岗，我悟了！！！

大家好，我是Yuki。周末跟一个在B站面试大模型算法实习岗的学员聊了整整两个小时，他说这场面试让他“一边冒汗一边开窍”。我让他把面试题完整复述了一遍，今天就把这场高质量的技术对话分享给大家。

说实话，这几道题问得是真有水平——不是那种背八股文能应付的，而是实打实考察你做项目的深度和思考力。

面试复盘：一场关于Agent的技术拷问
第一关：你现在做的Agent到底是什么？

面试官开门见山，没有废话。

如果你回答“就是一个能调用工具的AI”，那基本就凉了。面试官要听的是：你对自己做的事情有没有系统性的理解。

正确的打开方式是这样的：

“我负责的是一个面向科研场景的辅助Agent。它的核心定位不是替代研究者，而是解决科研流程中的三个痛点：文献检索碎片化、实验记录不规范、代码复现困难。我们把它设计成一个多角色协作系统，每个Agent负责科研环节中的一个具体任务。”

看出来差别了吗？先说痛点，再说方案，最后说架构。这是第一道门槛。

第二关：整个流程怎么设计的？

面试官追问流程设计，意思是在问：你有没有真正理解业务流和技术流的映射关系。

这里我画了一张图帮助大家理解：

面试官真正想看的是：你能不能把“一个Agent”拆解成“一组Agent”，并且说清楚它们之间怎么配合。

我那位学员答得不错，他提到了一个关键设计：反馈闭环。子Agent的执行结果会回流到主Agent，主Agent根据结果决定是直接输出还是启动修正流程。这个细节让面试官点了点头。

第三关：你说文献检索，是Deep Research吗？代码层面怎么实现的？

这个问题很刁钻。面试官在测试两件事：一是你能不能区分概念，二是你有没有真的写过代码。

先区分概念：Deep Research是一个完整的自主研究流程，包含假设生成、实验验证、结果整合；而你做的文献检索只是其中的一个模块。不要给自己脸上贴金，也不要妄自菲薄，准确表达即可。

再谈实现，这才是重头戏。这位学员当时是这么回答的：

“代码层面主要解决三个问题：检索策略、结果解析、质量评估。”

检索策略用的是查询扩展——用户输入‘LLM推理效率’，我们通过同义词库和领域本体扩展成3-5个检索式，并行访问arXiv、Semantic Scholar、PubMed等多个数据源。

结果解析麻烦的是PDF。我们用了一种分层解析方案：先用PyMuPDF快速提取文本和基础元数据，对包含公式和复杂表格的论文再调起Grobid做结构化解析。

质量评估这块，我们做了一个轻量级的相关性打分模型，基于SBERT，对标题和摘要做向量召回，过滤掉相关性低于阈值的结果。”

面试官接着问了一句：“那你们怎么处理检索结果的去重？”——这位学员说他们用了SimHash加上DOI精确匹配，双重去重。面试官明显对这个回答比较满意。

第四关：主Agent和子Agent共用同一个上下文吗？

这个问题考察的是对多Agent通信机制的理解。

正确答案是：分场景讨论，没有银弹。

这位学员的回答逻辑是：

共享场景：当多个Agents需要协作完成一个连贯任务时，比如文献调研→提炼观点→生成综述，共享上下文能避免信息断层。
不共享场景：当一个子Agent只负责独立子任务（比如纯代码生成）时，让它共享主Agent的完整历史只会引入噪声，降低性能，还容易泄露不该泄露的信息。

他最后总结了一句：“我们的方案是选择性共享——主Agent维护一个核心上下文窗口，子Agent只接收过滤后的相关片段。”这个回答有深度，说明他真的踩过坑。

第五关：讲一讲Tool Calling的具体实现

这道题是实打实的技术考察。大模型本身不能调用工具，它只能“表示想调用的意图”——这是很多新手容易搞混的点。

完整的实现流程是这样的：

技术细节上需要关注三层：

工具描述层：用OpenAI的Function Calling格式或Anthropic的Tool Use格式，把工具的名称、描述、参数schema传给大模型。
解析执行层：大模型返回的是一个JSON结构，你需要解析出tool名称和参数，然后去注册中心找到对应工具的真实函数，执行它。
结果回填层：工具执行的输出要重新喂给大模型，让它基于工具结果生成自然语言回复。

面试官可能会追问：“如果工具执行失败怎么办？”——这位学员回答了他们做了重试机制和降级策略。比如搜索引擎超时了，会尝试备用源；如果所有源都失败了，会明确告诉用户“工具调用失败，以下是根据模型自身知识生成的回答”，并且标注出来。这个细节很加分。

第六关：知道渐进式披露吗？解释一下

这个问题问得非常聪明。渐进式披露最初是UI/UX领域的概念，放到Agent场景里，指的是不要一次性给用户塞太多信息，而是按需、分层次地展示。

举几个例子你就明白了：

文献检索：先展示标题、作者、年份、摘要摘要，用户点开才显示全文解析结果和关键图表
代码生成：先展示代码的整体结构和核心函数，用户确认逻辑后再展开细节实现和依赖说明
多步推理：Agent先给出最终结论，用户追问“为什么”时才展开推理链

这位学员还举了一个反面案例：他们刚开始做Agent的时候，每次返回直接把几千字的完整分析全甩给用户，结果用户反馈“看着很炫酷但不知道怎么用”。后来改成分层展示，用户停留时长反而提升了。

面试官听完微微一笑——这说明你真的用过、踩过坑、优化过。

第七关：用LangGraph遇到的最大困难是什么？

这道题是压轴题。面试官不是在问你LangGraph的缺点，而是在问：你在复杂系统里解决问题的能力。

这位学员的回答很真诚，他说最大的困难是状态管理的不可预测性。

具体来说：

“LangGraph虽然提供了状态图的概念，但当你的Graph有十几个节点、几十条条件边时，状态在节点间的传递经常出现‘幽灵变量’——某些字段莫名其妙变成了None。我们排查了很久，最后发现是LangGraph在深拷贝状态时的bug。我们的解决方案是重新封装了状态传递层，所有跨节点的关键数据强制用JSON序列化后再反序列化，防止引用传递带来的副作用。”

他还补充了第二个困难：调试体验差。LangGraph没有像LangSmith那样好用的可视化调试工具，他们被迫自己写了一个状态日志记录器，每个节点进出都打印状态快照，才能追踪问题的根源。

这个回答好在哪里？承认了困难，但没有甩锅给框架，而是说了自己怎么解决和规避的。这就是面试官想看到的态度。

这场面试给我的启发

复盘完这七道题，我想说三点：

第一，大模型面试已经告别“八股文时代”。不会有人再问你“什么是自注意力机制”这种问题，因为答案GPT都能告诉你。面试官要考察的是：你在真实场景里做过什么、遇到了什么问题、怎么解决的。

第二，Agent是一个系统工程，不是模型调试。这七道题里有六道都在问系统设计、数据流、异常处理、状态管理——这些都是后端和架构的范畴。想做Agent的同学，建议补一补系统设计的基础。

第三，深度比广度重要一万倍。用过十个框架不如把一个框架的坑讲透。这位学员被追问LangGraph的时候，那种“我们踩过坑”的真实感，是任何面试技巧都没法伪装的。

如果觉得这篇文章对你有帮助，欢迎点赞、在看、转发。你的支持是我继续深挖高质量面经的最大动力。

# 面试官视角拆一份Agent项目简历，防止避坑！

> 程序员花海 | 06-25 09:54

**标签**：我的求职进度条、实习，投递多份简历没人回...、简历中的项目经历要怎么写、27届秋招备战进行时

**来源**：https://www.nowcoder.com/discuss/898495069127200768

---

面试官视角拆一份Agent项目简历，防止避坑！

大家好，最近筛校招简历时翻到一份同时包含多 Agent 大模型项目与分布式高并发后端项目的简历，两份项目技术栈覆盖面看着很完整，既有当下热门 AI Agent 方向，又有传统分布式后端实战，乍一看很符合 27 届复合型开发求职者的定位。


但以一线大厂后端 + 大模型业务线面试官的视角逐字拆解后，能发现里面藏了大量应届生写项目的典型误区，很多表述看似专业，实则经不起深挖，初筛阶段容易被 HR 标记存疑，二面技术面试官追问时极易翻车。今天结合这份简历原文，逐条把所有问题拆解清楚，同时点明大厂筛选的评判逻辑，对照自己的项目经历自查修正。

一、全局底层共性问题 两份项目统一存在的核心缺陷
1. 通篇只有技术功能描述，完全缺失个人量化产出，没有落地数据佐证

两份项目所有亮点段落，只写自己设计了什么架构、用到什么技术组件，全程无任何可量化业务指标、性能优化数值、业务承载规模。

大厂技术面试官看校招项目，第一优先级看落地价值，比如校园心卫多 Agent 系统，没有说明支撑多少在校用户、每日交互请求量、风险预警识别准确率、系统平均延迟优化幅度；探店宝高并发项目只写接口 P99 从 500ms 降到 100ms，缺少核心前置数据，没有项目承载峰值 QPS、优惠券秒杀峰值流量、缓存击穿拦截次数、ID 生成器每日生成订单量级。

没有业务规模作为基础，单纯给出优化数字会显得悬浮，面试官会默认项目只是本地 Demo，没有线上真实部署运行，直接降低项目可信度。

2. 职责边界模糊，全程使用研发通用描述，无法区分团队共建与个人独立落地

两份项目全部没有标注项目团队规模、自身负责范围，大量使用研发通用话术，无法证明是自己独立设计核心方案，还是仅跟着教程复现 Demo、在团队中做边角辅助开发。

校园心卫多 Agent 协同系统涉及 MCP 协议封装、三级动态 RAG 路由、多智能体调度整套完整架构，探店宝覆盖分布式限流、缓存一致性、分布式 ID 三大核心分布式难点，两套复杂架构全部堆砌在个人项目中，却不区分分工。大厂面试官会天然产生质疑，应届生很难独立完整落地两套高复杂度架构，大概率存在复制开源项目、照搬课程案例、团队多人协作却全部算作个人成果的情况。

3. 缺少项目交付形态与部署落地信息，分不清本地 Demo 和可线上访问工程

通篇没有提及项目部署方式、运行载体、交付对象，校园心卫没有说明是否上线校内测试环境、是否对接高校心理咨询真实业务；探店宝没有说明是否完成容器化部署、是否搭建压测环境验证高并发逻辑。

现在大厂校招评判项目有明确区分标准，本地单机跑通的 Demo 和支持多用户并发访问、具备异常兜底机制的线上工程含金量天差地别，简历完全不体现部署、测试、交付相关内容，会直接被归类为课程演示项目，工程能力得分大幅下滑。

二、校园心卫 大模型多 Agent 项目专属问题 大模型业务线面试官重点扣分点
1. 专业术语堆砌过度，概念耦合混乱，存在逻辑自相矛盾，极易深挖翻车

第一处硬伤，异步任务调度模块同时提及 MCP 协议封装原子化工工具链、消息队列解耦高风险处理，表述模糊。MCP 协议本身是面向大模型工具调用的标准化协议，原生不配套消息队列死信队列、重试告警整套分布式异步机制，简历没有说明自研改造逻辑，面试官追问 MCP 与 RabbitMQ 联动的适配方案、协议层如何兼容消息重试兜底逻辑，很难完整作答。

第二处硬伤，动态路由 RAG 设计三级图路由策略描述空洞，只说按需触发 RAG 检索降低 Token 消耗，没有说明路由判断依据、不同层级检索区分规则、Token 消耗具体下降比例，概念停留在表层名词堆砌，看不出自主设计思考，只会被判定为照搬 LangGraph 开源案例。

第三处硬伤，多 Agent 协同架构描述冗余笼统，提到 Supervisor 中枢调度、Memory、Knowledge、RiskGuardian、Counselor 多专家智能体，只说实现诉求拆解流转闭环，没有说明多智能体之间状态同步机制、智能体冲突处理方案、心理场景下对话记忆持久化逻辑。当下大量应届生照搬 LangGraph 官方多 Agent 案例，面试官高频深挖自定义调度逻辑，如果只是调用框架原生能力，这段亮点直接作废。

2. 业务场景结合薄弱，技术方案和高校心理咨询业务痛点脱节

项目背景点明痛点是心理咨询资源紧缺、危机干预滞后，但所有功能亮点全部聚焦底层架构技术实现，没有对应业务层面解决效果。

比如风险预警 Agent 没有说明如何识别自杀、抑郁等高风险对话、预警推送后联动校内辅导员的完整业务闭环；多专家 Agent 没有区分普通心理疏导、危机干预、陪伴安抚三种场景的调度差异化逻辑，单纯堆砌 Agent 架构技术，没有体现技术对业务痛点的解决价值，会被认为只是套壳大模型 Demo，没有真实业务思考。

3. 缺少大模型工程化关键内容，忽略校招 AI 方向核心考察点

目前大厂 AI Agent 校招非常看重大模型工程落地能力，这份项目完全缺失模型推理优化、向量库选型与分片、提示词工程迭代、多轮对话上下文窗口裁剪等核心考点。通篇只讲 Agent 调度架构，不涉及大模型层面的落地优化，技术体系存在明显短板，复合型求职者本该体现的大模型工程能力完全没有展现。

三、探店宝 分布式后端项目专属问题 后端架构面试官高频质疑点
1. 高并发优化描述存在逻辑漏洞，部分方案实现细节缺失，经不起场景追问
分布式令牌桶限流部分，采用 Redis+Lua 实现原子库存扣减，只提及一人一单校验，没有说明 Lua 脚本如何处理超卖极端场景、限流令牌桶动态扩容缩容策略；面试官会追问秒杀场景下热点商品缓存过期瞬间的流量冲击应对方案，简历无对应兜底设计，方案完整性不足。
消息可靠性投递仅说明唯一订单号防重，缺少生产者确认、消费者 ack 机制、消息幂等完整链路说明，只提单一防重手段，分布式消息一致性方案残缺，暴露分布式事务基础掌握不扎实。
缓存逻辑采用逻辑过期 + 异步重建，只说明容忍短暂数据不一致窗口，没有明确窗口时长、异步重建失败降级策略、热点缓存击穿兜底方案，对于电商优惠券秒杀场景，短暂不一致会直接引发超发问题，简历没有配套业务降级方案，架构设计存在明显缺陷。
2. 分布式 ID 方案描述片面，规避冲突的实现逻辑表述模糊

自定义 Redis 全局 ID 生成器，只说明时间戳偏移量 + 每日自增序列号，没有说明偏移量如何分配多服务节点、每日序列号溢出兜底策略、Redis 宕机时 ID 生成降级方案。分布式 ID 核心难点就是多实例冲突与中间件故障兜底，简历完全回避故障场景处理，面试官一眼就能看出只实现了单机可用基础版本，没有考虑分布式集群生产环境异常。

3. 业务价值弱化，高并发优化没有对应真实业务场景数据支撑

探店宝定位生活服务商户优惠券秒杀平台，全程没有说明优惠券峰值发放数量、活动峰值 QPS、缓存击穿拦截的无效请求量级、分布式 ID 支撑的日均订单总量。

单纯抛出 P99 接口耗时优化数值，没有业务流量作为支撑，无法证明这套分布式架构真正承接过高并发流量，在后端面试官眼中，等同于本地压测脚本模拟，没有线上业务落地经验。

四、简历优化核心修正思路 贴合大厂校招评判标准
所有项目补充前置基础信息：团队人数、个人独立负责模块、项目部署环境、业务承载基础数据，所有优化点配套量化落地指标，用业务规模支撑优化效果。
拆分技术方案与业务价值，每一段技术亮点先说明对应解决的业务痛点，再讲实现方案，最后附上量化优化结果，做到业务 - 技术 - 数据完整闭环。
精简冗余技术名词，删掉浅层框架调用类描述，重点突出自主改造、自研设计、异常兜底、故障降级等体现独立思考的内容，减少开源框架原生能力堆砌。
区分两套项目侧重点，AI Agent 项目侧重业务场景落地、大模型工程优化；分布式后端项目侧重高并发异常兜底、分布式一致性完整链路，补齐两套技术体系缺失的核心工程化内容。
明确区分团队协作与个人自研模块，避免两套高复杂度完整架构全部划归个人独立开发，降低面试官对项目真实性的质疑。

最后补充一句，27 届现在复合型开发求职赛道内卷程度持续走高，同时掌握后端分布式与 AI Agent 技术栈是加分项，但前提是项目内容经得住深挖。简历单纯堆砌复杂架构名词只会适得其反，大厂面试官筛选项目的核心逻辑永远是看你的独立思考、工程落地能力和真实业务产出，缺少这三点，再丰富的技术栈也很难拿到二面机会。

# 知乎后端开发-一轮面经

> 在北京开网约车的后端开发 | 06-12 21:52

**标签**：学历对求职的影响

**来源**：https://www.nowcoder.com/discuss/895065862061686784

---

知乎后端开发-一轮面经

【岗位描述】
公司：知乎
岗位：后端开发实习生-研发运营

【面试流程与题目】

腾讯会议-40min

自我介绍：常规流程，突出项目经验和对AI/大模型方向的兴趣。
Code Review（秒杀业务）：面试官给了一段秒杀下单的代码，让找问题。

主要考察并发安全和事务：库存扣减和订单创建没有做并发控制，容易超卖；一人一单校验和库存校验存在竞态条件；@Async 异步通知如果抛异常没有兜底处理；事务方法内部调用异步方法，可能导致事务还未提交就发送了通知。

3. SQL性能分析：给出建表语句、索引和三条查询 SQL，要求分析每一条的性能问题：

DATE(create_time) = '2026-06-01'：对字段做函数运算会导致索引失效，建议改为范围查询；
phone_number = 123456：隐式类型转换，字符串字段传了数字，索引失效；
多条件组合查询：考察联合索引的最左前缀匹配原则。

4. 算法手撕：

LCR 076：数组中的第K个最大元素（快速选择/堆）
977：有序数组的平方（双指针）

5. Java基础：Java锁的底层实现原理（AQS、CAS、synchronized升级过程）。

6. 分布式中间件：实际项目中有没有用过分布式锁？详细讲讲Redisson底层原理（看门狗机制、RedLock等）。

7. AI/大模型方向：

如果让你设计一个AI Agent系统，数据存储层你会选什么数据库？为什么？
向量数据库的Embedding Model一般用什么？

【个人心得】

Code Review 题是亮点也是难点 — 不是让你背八股，而是看你能不能从一个看似正常的代码里找出并发安全问题。

SQL 优化万变不离其宗 — 核心就两条：「索引最左前缀」+「不在索引列上用函数」。搞懂这两个，90% 的 SQL 优化题都能应对。

分布式锁要深入到源码级 — 只会 lock.tryLock() 是不够的，Watchdog 续期机制、Lua 脚本原子性、Pub/Sub 通知机制、RedLock 算法，这些都是高频追问点。

AI Agent 方向的面试要体现系统性思考 — 数据库选型不是简单说「用 Milvus」，而是要结合业务场景分析不同类型数据的存储需求，给出一个完整的存储架构设计。

"你平时写代码，用AI辅助吗？大概有多少比例是AI生成的？"

我的第一反应

这个问题听起来很简单，但我直觉告诉我：他没有在问我用没用AI，他在问我会不会用AI。

于是我没有急着回答比例，而是先问了一个问题："您问这个是想了解我日常的工作方式，还是想了解我对AI编程工具的看法？"

他笑了笑："你先说说。"

我的回答

"我用AI辅助编程大概有一年半了。现在的比例大概是……60%到70%是AI生成的代码。"

"但这个比例不重要，重要的是我在这60%里扮演什么角色。"

他来了兴趣："什么角色？"

"reviewer。"

我继续解释

"AI生成的代码，我会做三件事：

第一，确认逻辑正确。 AI有时候会写出语法正确但逻辑有问题的代码。比如它写了一个分布式锁的实现，看起来像那么回事，但释放锁的时候没有考虑超时场景。

第二，检查边界情况。 AI的测试覆盖率普遍不够，尤其是异常路径。我会补充边界case的测试。

第三，优化可读性。 AI写的代码有时候能用，但不好看——变量命名不规范、注释缺失、函数过长。我会做一次重构。"

"所以，AI负责生成，我负责保证质量。这个分工是我这一年半摸索出来的。"

他的追问

他听完，点点头，又问了一个问题：

"那你觉得，AI写代码和人工写代码，最大的区别是什么？"

我想了想：

"AI的优势是速度和信息量。它能在几秒钟内遍历整个GitHub，找最优解。但它的劣势是……它不知道什么是'对'，它只知道什么是'可能对'。"

"比如，我让AI写一个订单取消的接口。它能写出功能完整的代码。但它不会主动问我：'取消订单的时候，要不要发通知？要不要扣积分？要不要生成审计日志？'"

"这些业务逻辑的判断，必须人来做。"

第三轮追问

他接着问："如果让你设计一个AI编程工具的评价体系，你会怎么设计？"

这道题我没准备过，但思考了几秒之后，我说：

"第一，正确率。 这个不用多说，但正确率不能只看功能对不对，还要看边界情况。

第二，可读性。 AI生成的代码，别人能不能看懂？维护成本高不高？

第三，可验证性。 这是最关键的——AI生成的代码，能不能自己证明自己是对的？"

他眼睛亮了："展开说说可验证性。"

"比如，AI写了一个排序函数。它能不能在交付之前，自动跑一组测试用例，证明它对所有情况都有效？最近有个项目叫OpenSquilla，就是在解决这个问题——让AI写代码的时候，自动生成红绿回归证据链。"

【顺便提一嘴】技术大厂，前端-后端-测试，全国均有机会。待遇和稳定性都还不错~ 感兴趣可以试试~

最后一轮

他问："你觉得，未来AI会取代程序员吗？"

我笑了："会取代一部分。但取代的那部分，本来也不需要人来做。"

"比如，CRUD接口、简单脚本、数据清洗——这些AI已经能做得比大多数初级程序员好了。但架构设计、业务理解、系统稳定性保障……这些AI至少在短期内做不到。"

"所以，与其担心被AI取代，不如想想：我做什么事情，是AI做不到的？"

他听完，说了一句让我印象很深的话：

"你是我今天面过的所有人里，唯一一个能说清楚'人和AI的边界在哪里'的候选人。"

面试结果

面完第二天，收到了offer。

薪资比我预期的还高了一点——他们说，看中的是我的"AI思维"。

入职之后我才知道，这个技术总监之前在微软干了十年，做的是Copilot的项目。他在面试里问的那些问题，其实都是在筛选"知道怎么和AI协作"的程序员，而不是"会用Copilot写代码"的程序员。

我的想法

这次面试让我明白了一个道理：AI时代，程序员的核心竞争力不是"会写代码"，而是"会审查代码"。

以前我们说"代码是程序员的产品"。现在我们说"AI生成的代码 + 人的审查 = 程序员的产品"。

这个等式变了，评价标准也得跟着变。

回到最初的问题——我有多少代码是AI生成的？

答案是：60%是AI写的，但100%是我负责的。

# 面试要求半小时完成一个demo

> 犹豫的小狐狸刷了100道题 | 07-02 18:48

**标签**：我的求职进度条

**来源**：https://www.nowcoder.com/feed/main/detail/b81a69c8502143db93701451061b54c6

---

这个问题的根不在你代码写得慢，在于需求没对齐就开始动手了。

面试官说"Demo"，你理解成写个页面，他心里想的是一个能跑的产品。这中间的差距不是AI能帮你补的——AI也不知道他脑子里要什么。佩奇卡说得对，现在面试官默认你会拉AI一起拆需求，但拆需求的第一步不是让AI写代码，是你自己先把"做到什么程度"问清楚。

拿到这种一句话需求，别急着开编辑器。花两分钟反问：这个Demo要跑到什么程度？是页面原型就行，还是要有真实数据流转？要不要对接第三方接口？这步问清楚了，后面半小时才有方向。面试官说"不会问AI"大概率不是让你问AI怎么写代码，而是让你先把需求边界搞明白。

不过话说回来，半小时内要求完成一个带平台对接和岗位匹配的东西，这个预期本身偏高。如果面试官自己也没把需求讲清楚就怪你理解不到位，那走也不可惜。

# 何意味？招我来当实习生，但不给我活

> Cor09 | 07-02 18:27

**标签**：我的实习日记、我的实习日记

**来源**：https://www.nowcoder.com/feed/main/detail/879c0433c55e4e58b99b5a61f22606c0

---

你再要一次活，结果还是一样。问题不在你态度不够，在于你"要活"的方式是空的——"想多做点事"这句话，mentor听了不知道给你什么。

他不是不想给，是他手上没有一块能直接切给你的活。正式工干的事大部分需要业务上下文，新人接不了。能给你的通常是杂活，但杂活太少，他懒得切。

与其继续要，不如做一件事：看他正在做什么，提前帮他做前置准备。比如他在做某个功能改造，你先去调研两三种实现方案，做个对比给他看。这步不需要他给你权限，不需要他安排，你做完丢给他，他省了调研时间，自然就把你纳入流程了。

找工作勤劳小蜜蜂说的那种"自己把老员工的任务做了"是激进做法，能用但风险也高——容易让人觉得越界。做前置准备更稳，因为你做的东西是给他参考的，接不接由他决定，不会踩到谁的领地。

转正这件事，靠的要不到的活，得自己造出来。
