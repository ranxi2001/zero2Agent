# 业务 AI 工程分析

这场面试的特点是**从项目架构出发，逐层深挖工程细节**。面试官先问整体架构，再追问多 Agent 通信、并发、异步，最后考系统设计——是一场典型的“从宏观到微观”的追问链。

多 Agent 并发与通信部分问得很细，占了近三分之一的题量。另外 MCP、AI Coding 方法论、Skill 设计等偏新的方向也被考到，说明蚂蚁对实际工程落地能力的重视。

---

## Q1：介绍一个 Agent 项目的整体架构设计

**新手答**：“用大模型加上一些工具调用，再加个 RAG。”

**高手答**：

一个典型的 Agent 项目架构分四层：

```text
┌─────────────────────────────────────────────┐
│  接入层：用户输入解析、意图路由、权限校验       │
├─────────────────────────────────────────────┤
│  编排层：状态机 / DAG 驱动任务流转、Agent 调度  │
├─────────────────────────────────────────────┤
│  能力层：LLM 推理、工具调用、RAG 检索、记忆读写 │
├─────────────────────────────────────────────┤
│  基础设施层：向量库、数据库、消息队列、监控告警   │
└─────────────────────────────────────────────┘
```

接入层负责把用户请求标准化，编排层决定任务怎么拆、谁来做、做完怎么流转。能力层是各个 Agent 的实际执行单元——每个 Agent 有独立的 System Prompt、工具集和上下文窗口。基础设施层提供存储、检索、通信的底座。

关键设计决策：编排层用状态机还是 DAG，取决于任务是否有固定流程。固定流程用状态机（确定性强），探索性任务用 DAG 或 ReAct 循环（灵活性强）。

**考察点**：你能不能把 Agent 系统当作一个正经的软件架构来描述，而不是“模型 + 工具”两个词。

## Q2：多 Agent 架构中，每个 Agent 分别负责哪一块？怎么拆分的？

**新手答**：“一个负责理解，一个负责执行。”

**高手答**：

以一个会议安排 Agent 为例，拆成四个角色：

1. **意图解析 Agent**：从用户输入中提取结构化信息——参会人、时间范围、会议主题、优先级。输出是一个填好的 JSON 槽位
2. **日程查询 Agent**：调用日历 API 查所有参会人的空闲时段，做交集计算，输出候选时间列表
3. **冲突协调 Agent**：当没有完美时间段时，根据优先级规则（比如主持人优先、时长优先）给出推荐方案，必要时生成追问话术
4. **执行 Agent**：确认后调用日历 API 创建会议、发送通知、生成会议纪要模板

拆分的原则不是“按步骤拆”，而是**按失败模式拆**——意图解析错了需要重新理解，日程冲突需要重新协商，这两种失败的处理逻辑完全不同，混在一起会导致无效重试。

**考察点**：你拆 Agent 有没有明确的依据，还是随意切分。

## Q3：什么情况下用单 Agent，什么情况下用多 Agent？

**新手答**：“复杂任务用多 Agent。”

**高手答**：

判断维度有三个：

| 维度 | 用单 Agent | 用多 Agent |
|------|-----------|-----------|
| 能力边界 | 整个任务用同一套能力就能完成 | 涉及明显不同的专业能力（写代码 vs 审查 vs 测试） |
| 上下文污染 | 子任务之间上下文高度统一 | 不同子任务的中间状态混在一起会互相干扰 |
| 并行性 | 必须串行完成 | 多个子任务可以同时进行，拆开能降低总延迟 |

多 Agent 的隐性成本：通信开销、调度复杂度、一致性保障。如果整个任务虽然长但上下文统一，一个 Agent 加状态机通常更稳。不是越多 Agent 越高级，很多场景单 Agent 反而更可靠。

**考察点**：你选架构时有没有明确的判断框架，还是凭感觉。

## Q4：多 Agent 之间的数据传输或通信一般怎么做？

**新手答**：“把上一个 Agent 的输出传给下一个。”

**高手答**：

通信方式按场景选：

1. **直接传递**（最简单）：上游 Agent 的结构化输出直接作为下游的输入参数。适合顺序链式编排，延迟低但耦合紧
2. **共享黑板（Blackboard）**：所有 Agent 读写一个中央状态存储，每个 Agent 只写自己负责的字段。适合需要多个 Agent 看到全局状态的场景
3. **消息队列**：Agent 之间通过 Kafka / Redis Stream 异步通信，生产者和消费者解耦。适合异步并行场景，但引入了消息顺序和幂等性问题
4. **事件驱动**：某个字段更新后，自动触发订阅了该字段的下游 Agent。比轮询高效，但需要事件总线基础设施

关键约束：**Agent 之间传递结构化结论，不传原始推理过程**。传 JSON 消息体（带 `task_id`、`role`、`action`、`payload`），不传自然语言长文——方便解析、追踪、调试。

**考察点**：你有没有多组件通信的设计经验，能不能按场景选方案。

## Q5：多个 Agent 之间有并发的情况，怎么处理？

**新手答**：“用多线程跑。”

**高手答**：

Agent 并发分两类，处理方式不同：

**无依赖并发**（各做各的，互不影响）：
- 用 `asyncio.gather` 或线程池并行执行
- 每个 Agent 有独立的上下文窗口，互不污染
- 设置单个 Agent 的超时，某个超时不阻塞其他 Agent，用默认值兜底

**有依赖并发**（共享资源或有执行顺序约束）：
- 用 DAG 定义依赖关系，拓扑排序后按层并行——同一层的 Agent 并行，跨层串行
- 共享状态用乐观锁或版本号控制，防止并发写入互相覆盖
- 结果汇聚点设置 barrier，等所有上游 Agent 完成后再触发下游

```text
      ┌→ Agent A ─┐
Input ┤            ├→ 汇聚 → Agent D
      └→ Agent B ─┘
         Agent C ──→ （独立，不参与汇聚）
```

总延迟 = 最慢的那条路径。优化方向是识别关键路径，对关键路径上的 Agent 做性能优先配置。

**考察点**：你对并发编程的基础是否扎实，能不能把它映射到 Agent 场景。

## Q6：多个 Agent 同时操作数据库或文件，这种并发怎么处理？

**新手答**：“加锁。”

**高手答**：

加锁是对的方向，但要分场景选锁的粒度：

1. **数据库并发写入**：
   - 不同字段 → 各写各的，不需要锁
   - 同一字段 → 乐观锁（带版本号，CAS 更新）优先，冲突率低时性能好
   - 高冲突场景 → 分布式锁（Redis `SET NX EX`），但要设合理的锁超时，防止 Agent 挂了锁不释放
2. **文件并发写入**：
   - 不同文件 → 无冲突
   - 同一文件 → 用文件锁（`flock`）或改成“写临时文件 → 原子 rename”模式
   - 更好的方案：**避免多 Agent 写同一文件**——架构上做职责隔离，每个 Agent 只写自己的输出文件，最后由编排层合并

3. **读写分离**：大部分 Agent 只需要读共享状态，写的 Agent 很少。用读写锁（`RWLock`）而不是互斥锁，读操作不互相阻塞

核心原则：**能通过架构设计避免冲突，就不要在运行时用锁解决**。

**考察点**：并发控制的工程经验，以及“架构避免冲突优于运行时加锁”的设计思维。

## Q7：如果 Agent 是异步执行的，怎么做？

**新手答**：“用 async/await。”

**高手答**：

异步执行要解决三个问题：**启动、追踪、回调**。

1. **启动**：用 `asyncio.create_task` 或消息队列把任务提交到后台，主流程不阻塞。每个任务分配唯一 `task_id`
2. **追踪**：任务状态存到 Redis 或数据库，状态流转为 `pending → running → success / failed`。编排层通过 `task_id` 查状态，不轮询 Agent 本身
3. **回调**：任务完成后通过回调函数或事件通知编排层。如果是 HTTP 场景，用 webhook 回调；如果是进程内，用 Future/Promise 的回调机制

**异步场景的关键陷阱**：
- **超时控制**：每个异步任务必须有超时，否则一个卡住的任务会永远占资源
- **结果聚合**：多个异步 Agent 的结果在不同时间返回，需要一个聚合器等待所有结果或在超时后用已有结果继续
- **错误传播**：异步任务失败了，错误信息要能传回编排层，不能静默丢失

```python
async def run_agents(tasks):
    results = await asyncio.gather(
        *[asyncio.wait_for(agent.run(t), timeout=30) for t in tasks],
        return_exceptions=True  # 不让单个失败炸掉全部
    )
    return [r for r in results if not isinstance(r, Exception)]
```

**考察点**：异步编程基础，以及异步场景下的超时、错误处理、结果聚合。

## Q8：RAG 混合检索机制是怎么实现的？

**新手答**：“用向量搜索。”

**高手答**：

混合检索是指**同时用多路召回，再融合排序**：

```text
            ┌→ 向量召回（语义相似度，覆盖同义表达）
query ──────┼→ 关键词召回（BM25，覆盖精确匹配）
            └→ 标签/元数据召回（结构化过滤，如时间、类别）
```

三路各取 top-K，合并后用 **Reciprocal Rank Fusion（RRF）** 或 Re-ranking 模型做融合排序：

```text
RRF_score(doc) = Σ 1 / (k + rank_i(doc))
```

其中 `rank_i` 是文档在第 i 路召回中的排名，k 是平滑常数（通常取 60）。被多路同时召回的文档分数更高。

精排阶段用 Cross-Encoder 对 query-doc 对做细粒度打分，精度远高于召回阶段的双塔模型，但计算量大，所以只对 top-N 候选做精排。

为什么要混合而不是只用向量：向量召回擅长语义匹配（“怎么退货” ≈ “退款流程”），但对精确词（订单号、产品型号）很差。BM25 正好互补。

**考察点**：你对 RAG 检索链路的完整理解，不只是“用向量数据库”。

## Q9：Agent 项目里遇到幻觉问题怎么减少、规避？

**新手答**：“让模型不要编造信息。”

**高手答**：

幻觉治理分三个阶段：

**生成前**：
- Prompt 里明确指令“只基于以下证据回答，如果证据不足请说不知道”
- RAG 召回的证据做相关性过滤，低于阈值的不传给模型——垃圾证据比没有证据更危险
- 控制温度参数，生产环境用低 temperature（0.1-0.3）

**生成中**：
- 用 structured output / function calling 约束输出格式，减少自由发挥空间
- 关键字段（数字、日期、ID）要求模型从证据中直接引用，不能自己推算

**生成后**：
- **事实校验**：把模型输出中的关键声明和原始证据做一致性比对
- **自检机制**：让模型（或另一个模型）对自己的输出做一次审查——“你刚才的回答中，有没有原始材料里找不到依据的内容？”
- **人工兜底**：高风险操作（支付、删除、合同条款）的结果必须经人工确认

幻觉不可能完全消除，关键是**把它限制在可接受的范围内，且在它出现时能被发现**。

**考察点**：你对幻觉治理有没有系统性方案，还是只会说“提示词里写清楚”。

## Q10：Agent 系统中一般需要开发哪些工具 / function？

**新手答**：“搜索工具、计算工具。”

**高手答**：

工具按用途分五类：

| 类别 | 示例 | 设计要点 |
|------|------|---------|
| 信息获取类 | 搜索引擎、知识库查询、数据库查询、API 调用 | 返回值要做语义包装，不能把原始 JSON 直接丢给模型 |
| 操作执行类 | 发邮件、创建日程、提交工单、调用外部服务 | 必须有确认机制，高风险操作不能模型自己决定执行 |
| 数据处理类 | 计算器、数据格式转换、文件解析（PDF/Excel） | 确定性计算不要让模型做，用代码工具保证精确 |
| 代码执行类 | Python 沙盒、SQL 执行器 | 必须有安全沙盒，限制文件系统和网络访问 |
| 状态管理类 | 记忆读写、任务状态更新、上下文压缩 | 这类工具是给 Agent 自己用的，不直接面向用户 |

每个工具的接口设计要遵循三个原则：
1. **输入输出 schema 明确**：用 JSON Schema 定义参数类型和约束，模型才能正确调用
2. **错误信息对模型友好**：返回的错误不是 stack trace，而是“查询失败，原因是日期格式不对，请用 YYYY-MM-DD 格式重试”
3. **幂等性**：同一参数多次调用结果一致，防止模型重试导致重复操作

**考察点**：你开发工具时有没有体系化的分类思维和接口设计规范。

## Q11：self-refine / 自我修正，具体做过哪些修正策略？

**新手答**：“让模型检查自己的输出，有错就改。”

**高手答**：

自我修正不是简单的“再想一遍”，要针对不同类型的错误设计不同的修正策略：

1. **格式修正**：输出不符合要求的格式（JSON 解析失败、缺少必填字段）→ 把格式错误信息和期望 schema 一起反馈给模型，要求严格按格式重新生成
2. **事实修正**：输出和已知事实矛盾 → 把矛盾点明确指出（“你说的是 X，但数据库显示是 Y”），让模型基于正确事实重新推理
3. **逻辑修正**：推理链条有跳步或矛盾 → 要求模型先输出逐步推理过程（Chain-of-Thought），再检查每一步是否成立
4. **工具使用修正**：调用了错误工具或参数不对 → 把工具返回的错误信息连同可用工具列表一起反馈，要求模型重新选择

关键约束：**修正必须有轮次上限**（通常 2-3 次），超过就输出当前最优结果并标记置信度。无限循环修正是最常见的工程事故。

```text
修正流程：生成 → 校验 → 反馈错误 + 上下文 → 重新生成 → 再校验 → 通过/超限兜底
```

**考察点**：你做过的修正是“笼统地让模型反思”，还是针对不同错误类型有不同策略。

## Q12：如果 API 返回结果有字段缺失或冗余内容，怎么处理？

**新手答**：“让模型自己处理。”

**高手答**：

API 返回的脏数据不应该直接丢给模型处理——模型处理结构化数据不如代码可靠。正确做法是在**工具层加一个 semantic wrapper（语义包装层）**：

1. **字段校验**：工具返回后先做 schema 校验，缺失必填字段时触发重试或降级逻辑，不传给模型
2. **冗余过滤**：只提取模型当前任务需要的字段，把无关字段剥离。比如订单 API 返回了 50 个字段，模型只需要订单号、状态、金额三个
3. **语义转换**：把机器格式转成模型友好的自然语言描述。`"status": 2` → `"订单状态：已发货"`。模型理解自然语言的能力远强于理解枚举值
4. **缺失补全**：非必填字段缺失时，用默认值或“未提供”显式标注，不留空——空值会让模型猜测

```python
def wrap_order_response(raw):
    return {
        "order_id": raw.get("id", "未知"),
        "status": STATUS_MAP.get(raw.get("status"), "未知状态"),
        "amount": f"¥{raw.get('amount', 0):.2f}",
        "summary": f"订单 {raw['id']}，状态：{STATUS_MAP[raw['status']]}，金额：¥{raw['amount']:.2f}"
    }
```

**考察点**：你有没有意识到模型和外部 API 之间需要一个适配层。

## Q13：你对 MCP 了解多少？有没有写过 MCP Server？

**新手答**：“MCP 就是一种工具调用协议。”

**高手答**：

MCP（Model Context Protocol）是 Anthropic 提出的**模型与外部工具/数据源之间的标准化通信协议**。核心思想是把工具调用从“每个框架自己定义”变成“一个统一协议”——类似于 USB 对外设的标准化。

MCP 的三个核心概念：
1. **Tools**：可执行的操作（查数据库、发请求），模型主动调用
2. **Resources**：可读取的数据源（文件、数据库表），类似 REST 的 GET
3. **Prompts**：预定义的提示词模板，Server 端定义、Client 端复用

MCP Server 的实现本质上是一个暴露了特定接口的服务，用 JSON-RPC 通信。写一个 MCP Server 的核心步骤：
1. 定义工具的 name、description、input schema
2. 实现工具的执行逻辑
3. 通过 stdio 或 HTTP+SSE 暴露给 MCP Client

MCP 的价值不在于技术多先进，而在于**生态复用**——写一次 MCP Server，所有支持 MCP 的 Client（Claude、Cursor、各种 Agent 框架）都能直接用，不用每个框架写一套适配。

**考察点**：你对 Agent 工具生态的新趋势有没有关注，以及对协议标准化价值的理解。

## Q14：Agent 调工具失败了，一般怎么处理？

**新手答**：“重试。”

**高手答**：

工具失败分五类，处理策略不同：

| 失败类型 | 示例 | 处理策略 |
|---------|------|---------|
| 参数错误 | 日期格式不对 | 把错误信息反馈给模型，让它修正参数重试 |
| 超时 | API 响应超过 10s | 先重试一次，仍超时则切备用工具或降级返回缓存结果 |
| 权限不足 | Token 过期 | 自动刷新 Token 重试，刷新失败则上报用户 |
| 服务不可用 | 第三方 API 宕机 | 切换到备用数据源，无备用则明确告知用户“该服务暂时不可用” |
| 逻辑错误 | 查询条件本身不合理 | 不重试，而是让模型重新分析任务，调整策略 |

关键工程约束：
- **重试有上限**：同一工具最多重试 2 次，避免死循环
- **重试要变参数**：第二次调用必须和第一次不同（参数修正或策略调整），一模一样的参数重试是无效重试
- **失败信息对模型友好**：不返回 stack trace，返回结构化的错误描述 + 建议的下一步操作

**考察点**：你对工具失败有没有分类治理的意识，还是只会“重试”。

## Q15：开发 Agent 时，怎么管理它的 context？

**新手答**：“把对话历史都带上。”

**高手答**：

上下文管理的核心是**让模型在每一步看到恰好需要的信息，不多不少**。分四层：

1. **固定层**（每轮必带）：System Prompt + 当前任务的关键约束（预算、日期、用户偏好）。这些信息不从对话历史中“找”，而是提取出来作为固定注入
2. **滚动层**（最近几轮）：最近 3-5 轮对话原样保留，保证短期强相关
3. **摘要层**（历史压缩）：每 10 轮让模型生成结构化摘要——“用户目标是什么、进展到哪、遇到什么障碍”，替代原始历史消息
4. **按需层**（动态召回）：建一个对话索引，按主题和实体标记每段对话，需要时精准召回，而不是全量塞回去

另外，**工具返回值和中间结果要落库**，上下文里只保留一句“已完成 XX，结果存储在 task_123”。原始的 API 返回可能几千 token，但对后续步骤有用的可能只有两三个字段。

```text
上下文 = 固定层（System Prompt + 约束）
       + 滚动层（最近 N 轮）
       + 摘要层（历史压缩）
       + 按需层（动态召回）
```

**考察点**：你对上下文管理有没有分层设计，还是“全部带上”。

## Q16：做 AI Coding 时，有没有什么方法论？

**新手答**：“让 AI 写代码，然后自己改改。”

**高手答**：

AI Coding 的关键不是“让 AI 写”，是**怎么给 AI 足够的上下文让它写对**。方法论分三步：

**1. 先定规格再写代码**：
- 写清 PRD 或技术设计文档，明确输入输出、边界条件、异常处理
- 让 AI 先做方案设计（“你打算怎么实现”），对齐思路后再写代码
- 这一步花 20% 时间，能减少 80% 的返工

**2. 分模块逐步生成**：
- 不要一次让 AI 生成整个项目，拆成模块逐个生成
- 每个模块生成后立刻 review 和测试，问题发现越早修复成本越低
- 给 AI 提供已有代码的上下文（接口定义、类型声明），保证模块间一致

**3. 建项目级约束**：
- 用 CLAUDE.md / .cursorrules 等项目配置文件定义编码规范、技术栈约束、禁止事项
- 复用的操作封装成 Skill，一次调教好反复用
- Code review 不只看功能对不对，还要看 AI 有没有引入安全漏洞或过度工程

核心认知：AI Coding 不是“人被替代”，是**人从“写代码”变成“写规格 + 做决策 + 做审查”**。

**考察点**：你用 AI Coding 的深度——是随便聊几句，还是有系统化的工作流。

## Q17：上线前通过什么方式保证质量？

**新手答**：“Code review。”

**高手答**：

Code review 只是其中一环。完整的质量保障分四层：

1. **自动化测试**：
   - 单元测试覆盖核心逻辑
   - 集成测试验证 Agent 端到端流程——给定输入，检查最终输出和中间步骤是否符合预期
   - 回归测试：维护一个 golden test set，每次改动后跑一遍，防止改一个坑出两个

2. **评估体系**：
   - 定义评估指标：任务完成率、步骤效率、幻觉率、工具调用正确率
   - 用评估数据集批量跑，不是靠手动试几个 case
   - A/B 测试对比不同版本

3. **Code review**：
   - 重点看：Prompt 变更（改一个词可能影响全局行为）、工具接口变更、安全相关逻辑
   - AI 生成的代码更要仔细 review——AI 容易写出“看起来对但边界条件不对”的代码

4. **灰度发布**：
   - 新版本先对 5% 用户放量，监控核心指标无异常后逐步扩大
   - 设置自动回滚机制：错误率超过阈值自动切回旧版本

**考察点**：你对质量保障有没有体系化思维，还是只知道 code review。

## Q18：怎么通过多源信息判断内容可信度？

**新手答**：“看来源是否权威。”

**高手答**：

以判断一篇帖子的可信度为例，流程分三步：

1. **特征提取**：
   - 内容特征：帖子本身有没有夸大用词、是否包含可验证的事实性声明、有没有引用来源
   - 社交特征：评论区的态度分布（支持/质疑/中立比例），高赞评论是否在打脸原帖
   - 作者特征：历史发帖的可信度评分、是否有认证标识

2. **交叉验证**：
   - 用帖子里的关键声明去检索外部知识库，看是否有佐证或矛盾
   - 类似声明在其他平台有没有出现过，多源一致的可信度高

3. **综合打分**：
   - 多维特征做加权打分，输出一个可信度评级（高/中/低/存疑）
   - 评论区质疑比例超过阈值时自动降级
   - 最终由模型生成一段可信度分析说明，而不是只给一个数字

关键是**不能只看帖子本身**——评论区的“群体验证”信息往往比帖子内容本身更有判断价值。

**考察点**：你有没有多源信息融合的设计思路。

## Q19：AI Coding 过程中，怎么提升代码质量、降低模型“补全乱写”的概率？

**新手答**：“多检查几遍。”

**高手答**：

降低“乱写”的核心是**约束模型的决策空间**：

1. **给足上下文**：把相关的类型定义、接口声明、已有代码喂给模型。模型在有参考的情况下生成的代码一致性远高于凭空生成
2. **用项目级规则约束**：CLAUDE.md / .cursorrules 里定义技术栈、命名规范、禁止使用的库——模型遵守显式规则的能力很强
3. **拆小任务**：一次只让模型做一件事。“写一个完整的用户管理模块”容易乱写，“写一个根据 user_id 查询用户信息的函数，入参和返回值 schema 如下”就不容易偏
4. **类型系统兜底**：用 TypeScript 而不是 JavaScript，用 Pydantic 而不是裸 dict——类型检查能自动拦住一大类“乱写”
5. **Lint + 自动化检查**：pre-commit hook 跑 lint、格式化、类型检查，不过就不让提交

本质上，“乱写”不是模型能力问题，是**上下文不足 + 约束不够**。给够约束，模型的输出质量会有质的飞跃。

**考察点**：你对 AI 辅助编程的认知深度——是“模型不行”还是“我没给够信息”。

## Q20：Agent 的 Skill 机制怎么设计？怎么评测一个 Skill 是否合格？

**新手答**：“Skill 就是一个 Prompt 模板。”

**高手答**：

**Skill 设计**：

Skill 是一个**可复用的能力单元**，不只是 Prompt——它包含触发条件、Prompt 模板、工具依赖、输出格式规范：

```text
Skill:
  name: "代码审查"
  trigger: 用户提交代码或要求 review
  prompt_template: [包含审查维度、输出格式的结构化 Prompt]
  tools: [code_search, file_read, lint_check]
  output_schema: {issues: [...], severity: ..., suggestions: [...]}
```

好的 Skill 设计原则：职责单一（一个 Skill 做一件事）、输入输出 schema 明确、可组合（Skill A 的输出能作为 Skill B 的输入）。

**Skill 评测**：

1. **功能正确性**：准备一组 golden test cases（输入 + 期望输出），批量跑通过率
2. **边界测试**：空输入、超长输入、格式异常输入——看 Skill 会不会崩或输出垃圾
3. **一致性测试**：同一输入跑 10 次，输出的核心字段是否稳定（允许措辞不同，但结论不能矛盾）
4. **性能基线**：记录平均延迟和 Token 消耗，建立基线，后续迭代不能明显退化

评测不是一次性的——每次改 Prompt 或换模型后都要重跑，防止回归。

**考察点**：你把 Skill 当“模板”还是当“工程组件”来设计。

## Q21：系统设计——如果要做一个微信小程序排查问题的 Agent，怎么设计？

**新手答**：“用 RAG 搜索历史问题记录。”

**高手答**：

分三层设计：

**第一层：问题理解与分类**

用户描述通常很模糊（“小程序白屏了”），需要先做结构化：
- 意图分类：是代码 bug、配置问题、性能问题、还是框架本身的问题
- 信息补全：追问关键信息——设备型号、微信版本、复现步骤、报错信息
- 对于“白屏”这种高频问题，用预置规则直接走快速排查路径，不进入通用流程

**第二层：多路诊断**

```text
              ┌→ 日志分析 Agent：解析用户提供的错误日志，提取关键异常
问题描述 ──────┼→ 知识库检索 Agent：RAG 检索历史问题库 + 官方文档
              └→ 代码分析 Agent：如果用户提供了代码片段，做静态分析
```

知识库检索不是简单的关键词匹配——要用混合检索（向量 + BM25），且按问题类型做分库（配置类问题查配置库，API 类问题查 API 文档库）。

**第三层：框架级问题的处理**

如果排查发现不是业务代码问题，而是 JS 框架本身的 bug：
1. Agent 检索框架的 changelog 和已知 issue 列表，匹配是否为已知 bug
2. 如果是已知 bug → 给出 workaround 方案和官方修复版本
3. 如果是未知 bug → Agent 无法自主修复，转为**辅助模式**：帮用户整理复现步骤、最小化复现代码、生成 issue 模板，引导用户提交到官方仓库
4. 关键原则：**Agent 要知道自己的能力边界**——框架级 bug 不是 Agent 能修的，但 Agent 能帮用户高效地定位和上报

```text
诊断结果 → 业务代码问题 → 给出修复建议
        → 配置问题     → 给出正确配置
        → 框架 bug     → 匹配已知 issue → 给 workaround
                       → 未知 bug      → 辅助整理复现 + 生成 issue 模板
```

**考察点**：系统设计能力，以及你能不能识别 Agent 的能力边界——不是什么问题都能自动修复。
